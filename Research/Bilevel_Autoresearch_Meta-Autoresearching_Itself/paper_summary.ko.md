# 논문 요약

## 한 줄 요약

`Bilevel Autoresearch`는 LLM이 실험 후보를 고르는 내부 autoresearch 루프를 그대로 두지 않고, 외부 루프가 내부 루프의 탐색 메커니즘 자체를 분석하고 Python 코드로 생성해 주입하는 방법을 제안한다.

## 초급 독자용 먼저 보기

이 논문을 처음 읽을 때는 모든 용어를 한 번에 이해하려고 하지 않아도 된다. 핵심은 다음 네 문장이다.

1. 기존 자동 연구 루프는 "하나 바꿔 보고, 좋아지면 유지하고, 아니면 버리는" 방식이다.
2. 이 방식은 같은 종류의 후보를 계속 고르는 버릇이 생길 수 있다.
3. 논문은 후보 자체가 아니라 후보를 고르는 방식, 즉 탐색 메커니즘을 바꾸는 외부 루프를 추가한다.
4. 실험에서는 이 외부 루프가 반복 탐색을 끊고 중요한 방향을 찾으면서 더 큰 성능 개선을 만들었다.

짧게 말하면, 이 논문은 "자동 실험기가 실험 설정만 고치는 것이 아니라, 자기 자신의 실험 습관도 고치게 하자"는 아이디어를 다룬다.

## 꼭 알아야 할 말

| 용어 | 쉬운 설명 |
| --- | --- |
| LLM | ChatGPT 같은 대형 언어 모델 |
| 후보 | 다음에 시도할 설정 변경 |
| 탐색 | 여러 후보를 시험하며 좋은 설정을 찾는 과정 |
| 루프 | 제안, 실행, 평가를 반복하는 흐름 |
| 메커니즘 | 후보를 어떻게 만들고, 어떻게 고르고, 어떻게 버릴지 정하는 규칙 |
| 주입 | 새 규칙을 실행 중인 프로그램에 연결하는 것 |

## 문제의식

기존 autoresearch는 보통 다음 루프를 반복한다.

1. 현재 코드와 결과를 읽는다.
2. LLM이 하이퍼파라미터나 코드 변경을 제안한다.
3. 실험을 실행한다.
4. 검증 지표가 좋아지면 유지하고, 아니면 버린다.

이 방식은 작업 성능을 개선할 수 있지만, 탐색 방식 자체는 사람이 설계한 고정된 절차에 묶여 있다. 논문은 여기서 한 단계 더 나아가 "탐색하는 방법도 연구 대상이 될 수 있는가?"를 묻는다.

예를 들어 어떤 실험기가 매번 learning rate만 조금씩 바꾸고 batch size는 제대로 보지 않는다고 하자. 이때 learning rate 값을 더 많이 바꾸는 것은 Level 1의 일이다. "이 실험기는 너무 한 변수만 본다. 앞으로는 다른 변수도 강제로 보게 하자"라고 규칙을 바꾸는 것은 Level 2의 일이다.

## 핵심 구조

논문은 세 수준을 구분한다.

| 수준 | 역할 | 바꾸는 대상 |
| --- | --- | --- |
| Level 1 | 내부 autoresearch | 훈련 설정, 하이퍼파라미터, 후보 변경 |
| Level 1.5 | 외부 전략 조정 | freeze/unfreeze, 탐색 가이드, 집중할 파라미터 |
| Level 2 | 메커니즘 연구 | 내부 루프의 탐색 로직, 제안 방식, 업데이트 구조 |

중요한 차이는 Level 1.5와 Level 2 사이에 있다. Level 1.5는 기존 탐색 규칙 안에서 어떤 파라미터를 볼지 조절한다. Level 2는 내부 루프가 후보를 생성하고 선택하는 구조 자체를 바꾼다.

기술적으로는 Level 1.5도 외부 루프처럼 보이지만, 논문에서 더 중요한 novelty는 Level 2다. Level 2는 단순히 "이번에는 이 파라미터를 보라"라고 지시하는 수준이 아니라, 반복을 막는 tabu search나 직교 탐색 같은 새 탐색 규칙을 코드로 만들어 붙인다.

## Level 1: 내부 루프

내부 루프는 Karpathy의 GPT pretraining benchmark 위에서 동작하는 표준 autoresearch 루프다.

- LLM이 현재 `train.py`, 편집 가능한 파라미터, freeze 목록, 전략 가이드를 읽는다.
- 후보 변경과 가설을 제안한다.
- 300초 훈련을 실행한다.
- `val_bpb`가 낮아지면 변경을 유지하고, 아니면 버린다.
- 각 repeat는 30 iteration으로 고정한다.

여기서 `val_bpb`는 검증 성능 지표다. 이 논문에서는 값이 낮아질수록 더 좋은 결과로 본다.

## Level 1.5: 전략 조정 루프

Level 1.5는 5번의 내부 iteration마다 실행된다.

- 개선 없이 반복 제안된 파라미터를 freeze한다.
- 새로운 영역으로 이동했다고 판단되면 일부 파라미터를 unfreeze한다.
- 덜 탐색된 파라미터를 우선하라는 guidance string을 내부 루프에 넣는다.

이 단계는 탐색 다양성을 늘릴 수 있지만, proposal 생성 로직이나 keep/discard 규칙은 바꾸지 못한다.

쉽게 말해 Level 1.5는 "그 변수는 잠깐 보지 말고 다른 변수를 보자"라고 말할 수 있다. 하지만 "후보를 고르는 알고리즘을 tabu search로 바꾸자"라고 실행 구조 자체를 바꾸지는 못한다.

## Level 2: 메커니즘 연구와 코드 주입

Level 2는 외부 루프가 내부 루프의 코드와 trace를 읽고 새 탐색 메커니즘을 Python 코드로 생성하는 단계다.

논문에서 Level 2 세션은 네 단계로 구성된다.

1. Explore: runner 코드와 trace를 읽고 인접 분야의 메커니즘 후보를 탐색한다.
2. Critique: 반복 제안, 파라미터 고착 같은 실패 모드에 맞는 후보를 평가한다.
3. Specify: class name, method signature, runner 통합 지점을 명세한다.
4. Generate: 실행 가능한 Python 코드를 생성하고 runner 패치를 만든다.

생성된 코드는 `importlib`로 동적 로딩 검증을 거친다. import가 성공하면 활성화하고, 실패하면 백업한 원본 runner로 되돌린다.

이 검증 단계가 중요한 이유는 Level 2가 만든 코드가 항상 안전하거나 실행 가능하다고 보장할 수 없기 때문이다. 실제 논문에서도 외부 라이브러리 의존성 때문에 되돌려진 메커니즘이 있었다.

## 실험 설계

논문은 네 그룹을 비교한다.

| 그룹 | 활성 수준 | 의미 |
| --- | --- | --- |
| A | Level 1 | 순수 내부 autoresearch |
| B | Level 1 + 1.5 | 전략 조정만 추가 |
| C | Level 1 + 1.5 + 2 | 전체 bilevel 구성 |
| D | Level 1 + 2 | 전략 조정 없이 메커니즘 연구만 추가 |

모든 그룹은 같은 LLM, 같은 GPU 조건, 같은 baseline, 300초 훈련 예산, 30 iteration 예산을 사용한다. 각 그룹은 3회 반복된다.

## 주요 결과

논문 및 공개 저장소의 ablation 요약에 따르면 평균 `Delta val_bpb`는 다음과 같다. 더 음수일수록 개선이 크다.

| 그룹 | 평균 Delta val_bpb | 해석 |
| --- | --- | --- |
| A | -0.009 +- 0.002 | 내부 루프만으로는 작지만 안정적인 개선 |
| B | -0.006 +- 0.006 | 전략 조정은 다양성을 늘렸지만 신뢰할 만한 추가 개선 없음 |
| C | -0.045 +- 0.030 | Level 2 포함 시 Group A 대비 약 5배 개선 |
| D | -0.034 +- 0.031 | 큰 개선 가능성은 보이나 ablation 해석이 덜 깨끗함 |

초급 독자는 먼저 A와 C만 비교해도 된다. A는 기존 방식이고, C는 전략 조정과 메커니즘 변경을 모두 넣은 방식이다. C의 평균 개선 폭이 훨씬 크다는 점이 논문의 핵심 결과다.

Level 2가 생성한 대표 메커니즘은 다음과 같다.

| 메커니즘 | 분야 | 역할 |
| --- | --- | --- |
| Tabu Search Manager | 조합 최적화 | 최근 실패한 파라미터 영역 재방문 방지 |
| Multi-Scale Bandit Proposer | 온라인 러닝, multi-armed bandit | 파라미터 선택에서 exploration/exploitation 균형 |
| Systematic Orthogonal Exploration | 실험계획법 | 한 파라미터에 고착되지 않고 직교 방향 탐색 |
| GP Regressor | Bayesian optimization | `sklearn` 미설치로 revert |

## 왜 개선되었는가

핵심은 `TOTAL_BATCH_SIZE` 감소 방향을 발견했는지 여부다. Group A와 B는 기본 LLM prior 때문에 batch size를 늘리는 방향을 먼저 시도하고 실패한 뒤, 그 반대 방향을 제대로 탐색하지 못했다. Group C의 Level 2 메커니즘은 반복 탐색을 끊고 직교 방향이나 금지 목록 기반 탐색을 강제하면서 batch size 감소를 찾았다.

논문은 이 결과를 "더 강한 모델을 쓴 효과"가 아니라 "같은 모델을 다른 bilevel 구조로 배치한 효과"로 해석한다.

이 대목이 중요하다. 외부 루프가 더 똑똑한 모델을 쓴 것이 아니라, 같은 모델을 다른 역할에 배치해 탐색 절차를 바꾸었다는 주장이다.

## 한계와 비판적 읽기

- 샘플 수가 작다. 그룹당 3회 반복은 통계적으로 충분하지 않다.
- benchmark가 하나뿐이다. 다른 모델 크기, 작업, 예산으로 일반화된다는 증거는 부족하다.
- Group D는 생성 메커니즘 패치가 일부 정상 적용되지 않아 Level 2 단독 효과를 깨끗하게 분리하지 못한다.
- Python 코드 주입은 강력하지만 취약하다. import 검증과 rollback만으로 안정성이나 수렴을 보장하지 않는다.
- 외부 dependency 문제가 있다. 실제로 GP Regressor는 `sklearn` 미설치로 되돌려졌다.
- prompt가 후보 분야를 제안했기 때문에 완전히 무제약적인 메커니즘 발견은 아니다.
- recursive self-improvement는 가능성으로 제시되었을 뿐, 이 논문에서 실증된 결과는 아니다.

따라서 이 논문을 읽을 때 "AI가 무한히 자기 개선을 증명했다"라고 해석하면 과하다. 더 정확한 해석은 "한 benchmark에서 외부 루프가 내부 탐색 루프의 행동을 개선할 수 있음을 보였다"이다.

## 학습 포인트

이 논문을 읽을 때는 "agent가 자기 자신을 개선했다"는 큰 표현보다, 더 구체적인 질문에 집중하는 편이 좋다.

- 내부 루프가 무엇을 최적화하는가?
- 외부 루프는 내부 루프의 어느 부분을 바꾸는가?
- 바뀐 메커니즘이 실제로 다음 후보 분포를 바꾸는가?
- 검증, rollback, dependency 제약은 충분한가?
- 같은 효과를 더 단순한 탐색 알고리즘으로 얻을 수 있는가?

## 초급 독자 이해 체크

아래 질문에 답할 수 있으면 큰 흐름은 이해한 것이다.

1. Level 1은 무엇을 바꾸는가?
2. Level 2는 왜 단순 하이퍼파라미터 튜닝과 다른가?
3. 논문에서 batch size 감소 방향을 찾는 것이 왜 중요했는가?
4. 생성 코드를 바로 쓰지 않고 import 검증과 rollback을 하는 이유는 무엇인가?

## 선수 지식과 기호

필요한 선수 지식은 2D 합성곱, 완전 연결층, 확률, 평균제곱오차, 교차 엔트로피, 미분의 기초다. 이 글에서는 좌상단이 원점이고 $x$는 오른쪽, $y$는 아래쪽으로 증가한다고 약속한다.

| 기호 | 뜻 | 범위 또는 shape |
| --- | --- | --- |
| $N$ | batch 크기 | 양의 정수 |
| $S$ | 한 축의 grid cell 수 | YOLOv1은 $S=7$ |
| $B$ | 셀마다 예측하는 box predictor 수 | YOLOv1은 $B=2$ |
| $C$ | 클래스 수 | VOC는 $C=20$ |
| $(r,c)$ | 셀의 행·열 인덱스 | $0 \le r,c < S$ |
| $(t_x,t_y)$ | 셀 내부 중심 offset | 보통 $[0,1]$로 해석 |
| $(w,h)$ | 이미지 전체에 대해 정규화한 너비·높이 | $[0,1]$ |
| $q$ | confidence | 객체 존재와 위치 품질의 결합 표적 |
| $p_c$ | 셀의 조건부 class 확률 | $P(c\mid\mathrm{object})$ |
| $\operatorname{IoU}$ | 두 box의 교집합/합집합 넓이 | $[0,1]$ |

좌표 표현은 반드시 이름과 함께 적는다. 이 글의 `cxcywh`는 중심 $x$, 중심 $y$, 너비, 높이이고, `xyxy`는 좌상단 $x_1,y_1$과 우하단 $x_2,y_2$다. 네 숫자만 보고 두 형식을 구별할 방법은 없다.

## 1. 직관: 분류기를 공간적으로 복제한다

이미지 분류기는 이미지 하나를 클래스 벡터 하나로 줄인다. 객체 검출기는 객체의 종류뿐 아니라 위치와 개수도 알려야 한다. YOLOv1은 이 문제를 다음처럼 고정 크기 회귀 문제로 바꿨다.

1. 이미지를 $S \times S$ 셀로 나눈다.
2. 객체 중심이 들어간 셀이 그 객체를 담당한다.
3. 각 셀은 $B$개 box와 한 벌의 class 분포를 예측한다.
4. 모든 셀의 예측을 한 텐서로 동시에 출력한다.
5. confidence와 class score로 후보를 거르고 NMS로 중복을 줄인다.

핵심은 "box가 셀 안에 완전히 들어가야 한다"가 아니다. 객체의 중심이 어느 셀에 있는지가 책임 셀을 정한다. box는 셀 경계를 넘어갈 수 있다.

### 분류와 검출의 shape 차이

| 작업 | 대표 출력 | 의미 |
| --- | --- | --- |
| 이미지 분류 | $N \times C$ | 이미지마다 class score 한 벌 |
| YOLOv1 검출 | $N \times S \times S \times (5B+C)$ | 셀마다 box $B$개와 class score 한 벌 |

VOC 설정에서는 마지막 차원이 다음과 같다.

$$
5B+C = 5\times2+20=30
$$

따라서 출력은 $N \times 7 \times 7 \times 30$이고 이미지 한 장당 실수 개수는 다음과 같다.

$$
7\times7\times30=1470
$$

여기서 흔한 오해가 하나 있다. $B=2$라고 해서 한 셀이 서로 다른 객체 두 개의 class를 독립적으로 표현하는 것은 아니다. box는 두 벌이지만 class 분포는 셀마다 한 벌뿐이다. 같은 셀 안에 중심이 들어온 서로 다른 클래스의 객체 둘은 이 표현에서 충돌한다.

## 2. 원본에서 바로잡을 점

원본의 교육적 비유는 큰 흐름을 잡는 데 유용하지만, 구현 계약으로 옮기려면 다음을 분명히 해야 한다.

| 원본 설명 | 정확한 해석 또는 수정 |
| --- | --- |
| 셀마다 $B$개 상자와 $C$개 class 확률을 예측 | 맞다. 단, class 확률 $C$개는 box마다가 아니라 셀마다 한 벌이다. |
| 손실 합의 범위가 $i=0$부터 $S^2$, $j=0$부터 $B$ | 0-based 구현에서는 $0\le i<S^2$, $0\le j<B$다. 끝점을 포함하면 한 항씩 초과한다. |
| feature vector 1024개를 `Linear(1024, 1470)`에 입력 | 설명용 축약이다. 원 논문의 마지막 convolution feature는 공간축까지 펴서 fully connected layer로 들어간다. 단일 1024-vector와는 다른 모델이다. |
| "격자당 상자 수가 한정되어 작은 물체를 놓침" | 더 직접적인 병목은 셀마다 class 분포가 한 벌이고 객체 중심 책임이 셀 단위라는 점이다. $B$를 늘리는 것만으로 같은 셀의 여러 객체 충돌을 해결하지 못한다. |
| Focal Loss를 한 연구자의 발견으로 설명 | Part 3에서는 논문 저자 전체와 실험 맥락을 기준으로 다룬다. 단일 인물의 수학적 증명으로 보는 것은 부정확하다. |
| 예제에서 모든 표본에 같은 `alpha`를 곱함 | 표준 binary focal loss의 $\alpha_t$는 양성·음성에 따라 달라진다. Part 3에서 수정 구현한다. |

또한 YOLOv1, 최신 YOLO 계열, SSD, RetinaNet은 같은 출력 형식이나 같은 손실을 쓰지 않는다. "YOLO"라는 이름을 최신 구현 전체의 단일 계약처럼 사용하면 디코더와 학습 target을 잘못 연결하기 쉽다.

## 3. 출력 텐서를 끝까지 추적하기

### 3.1 NHWC 논리 뷰

설명을 위해 마지막 출력을 다음 논리 shape로 둔다.

$$
\mathbf{Y}\in\mathbb{R}^{N\times S\times S\times(5B+C)}
$$

한 셀의 마지막 축은 다음처럼 자른다.

$$
[t_x^{(1)},t_y^{(1)},w^{(1)},h^{(1)},q^{(1)},\ldots,
t_x^{(B)},t_y^{(B)},w^{(B)},h^{(B)},q^{(B)},p_1,\ldots,p_C]
$$

PyTorch convolution은 보통 `NCHW`를 쓰므로 실제 head 직후에는 `N x (5B+C) x S x S`일 수 있다. 손실 코드가 `N x S x S x (5B+C)`를 기대한다면 `permute(0, 2, 3, 1)`이 필요하다. `view`만으로 축 순서를 바꿀 수는 없다.

### 3.2 예제 shape

배치 2개, $S=7$, $B=2$, $C=3$인 축소 문제를 보자.

| 단계 | shape | 비고 |
| --- | --- | --- |
| 입력 이미지 | `2 x 3 x 224 x 224` | PyTorch `NCHW` |
| backbone feature | `2 x 64 x 7 x 7` | 예시 backbone |
| `1x1 Conv` head | `2 x 13 x 7 x 7` | $13=5\times2+3$ |
| `permute` 후 | `2 x 7 x 7 x 13` | 셀 단위 파싱이 쉬움 |
| box view | `2 x 7 x 7 x 2 x 5` | 마지막 5는 `tx,ty,w,h,obj` |
| class logits | `2 x 7 x 7 x 3` | box가 아니라 셀마다 한 벌 |

`reshape(N, S, S, B, 5)`를 전체 13채널에 적용하면 실패해야 정상이다. class 채널을 먼저 분리하지 않았기 때문이다.

## 4. 셀 좌표를 이미지 좌표로 디코딩하기

셀 $(r,c)$에서 예측한 $(t_x,t_y,w,h)$를 이미지 정규화 중심 좌표 $(x_c,y_c)$로 바꾸면 다음과 같다.

$$
x_c=\frac{c+t_x}{S},\qquad y_c=\frac{r+t_y}{S}
$$

YOLOv1의 $w,h$는 이미지 전체 크기에 상대적인 값으로 해석한다. `cxcywh`에서 `xyxy`로 바꾸면 다음과 같다.

$$
x_1=x_c-\frac{w}{2},\qquad y_1=y_c-\frac{h}{2}
$$

$$
x_2=x_c+\frac{w}{2},\qquad y_2=y_c+\frac{h}{2}
$$

추론 결과를 이미지 경계로 자를 때는 $[0,1]$에 clip할 수 있다. 그러나 학습 손실을 계산하기 전에 무조건 clip하면 경계 밖 예측의 오차와 gradient가 사라질 수 있으므로 시점을 구분한다.

### 4.1 수작업 예제

$S=7$, 셀 $(r,c)=(2,3)$, $(t_x,t_y,w,h)=(0.25,0.50,0.20,0.40)$라고 하자.

$$
x_c=\frac{3+0.25}{7}=0.4642857,\qquad
y_c=\frac{2+0.50}{7}=0.3571429
$$

따라서 다음 box를 얻는다.

$$
(x_1,y_1,x_2,y_2)
=(0.3642857,0.1571429,0.5642857,0.5571429)
$$

입력이 $448\times448$이라면 대략 $(163.2,70.4,252.8,249.6)$ pixel이다. 정규화 좌표에서 pixel 좌표로 갈 때 $x$에는 이미지 너비, $y$에는 이미지 높이를 곱한다.

### 4.2 NumPy 교차 검증

다음 코드는 **실행 가능한 검증용 예제**다.

```python
import numpy as np


def decode_cell(cell_rc, box, grid_size):
    row, col = cell_rc
    tx, ty, width, height = np.asarray(box, dtype=np.float64)
    cx = (col + tx) / grid_size
    cy = (row + ty) / grid_size
    return np.array([
        cx - width / 2.0,
        cy - height / 2.0,
        cx + width / 2.0,
        cy + height / 2.0,
    ])


actual = decode_cell((2, 3), (0.25, 0.50, 0.20, 0.40), 7)
expected = np.array([0.3642857143, 0.1571428571, 0.5642857143, 0.5571428571])
np.testing.assert_allclose(actual, expected, rtol=0.0, atol=1e-9)

pixels = actual * np.array([448.0, 448.0, 448.0, 448.0])
np.testing.assert_allclose(pixels, [163.2, 70.4, 252.8, 249.6], atol=1e-9)
print("numpy decode passed:", actual)
```

## 5. IoU와 책임 predictor

한 책임 셀 안에도 $B$개 predictor가 있다. YOLOv1 학습에서는 정답 box와 IoU가 가장 큰 predictor 하나가 좌표와 object confidence 학습을 담당한다. 이 선택은 class label에 따른 것이 아니다.

두 `xyxy` box $A,B$의 교집합 너비와 높이는 다음과 같다.

$$
w_I=\max\left(0,\min(x_2^A,x_2^B)-\max(x_1^A,x_1^B)\right)
$$

$$
h_I=\max\left(0,\min(y_2^A,y_2^B)-\max(y_1^A,y_1^B)\right)
$$

교집합과 합집합 넓이는 다음과 같다.

$$
I=w_Ih_I,\qquad U=|A|+|B|-I
$$

$$
\operatorname{IoU}(A,B)=\frac{I}{\max(U,\varepsilon)}
$$

$\varepsilon$은 0 넓이 box나 잘못된 입력으로 인한 0 나눗셈을 막는다. 실무에서는 `x2 >= x1`, `y2 >= y1` 계약도 검사해야 한다.

### 책임 할당의 비미분성

`argmax(IoU)`는 이산 선택이므로 책임 mask 자체를 미분하지 않는다. 선택된 predictor의 좌표와 confidence에는 gradient가 흐르지만, 어떤 predictor를 선택할지 결정하는 경로는 보통 target assignment로 취급한다. 구현에서는 assignment용 IoU를 `detach()`해 의도를 드러내는 편이 안전하다.

## 6. confidence와 class score

원 논문의 confidence target은 다음 의미를 갖는다.

$$
q=P(\mathrm{object})\times\operatorname{IoU}(\mathrm{pred},\mathrm{truth})
$$

객체가 없는 predictor의 표적은 0이다. 객체를 맡은 predictor에서는 $P(\mathrm{object})=1$이므로 target confidence가 IoU가 된다. 추론 시 클래스 $c$의 box score는 다음과 같다.

$$
s_c=p_c\times q
$$

조건부 확률 해석을 그대로 따르면 다음과 같이 읽을 수 있다.

$$
P(c\mid\mathrm{object})P(\mathrm{object})\operatorname{IoU}
=P(c)\operatorname{IoU}
$$

이 score는 보정된 확률이라고 보장되지 않는다. ranking과 threshold를 위한 결합 점수에 가깝다. 배포 환경에서 `0.5` 같은 threshold를 무조건 복사하지 말고 validation set의 precision-recall trade-off로 정한다.

## 7. YOLOv1 손실을 다섯 항으로 분해하기

셀 인덱스는 $0\le i<S^2$, predictor 인덱스는 $0\le j<B$로 둔다. $\mathbb{1}^{\mathrm{obj}}(i,j)$는 predictor $(i,j)$가 객체를 책임질 때 1이고, $\mathbb{1}^{\mathrm{noobj}}(i,j)$는 책임지지 않을 때 1이다. 아래 블록 수식에서는 같은 기호의 셀·predictor 인덱스를 아래첨자로 쓴다.

### 7.1 중심 좌표 손실

$$
L_{xy}=\lambda_{\mathrm{coord}}
\sum_{i=0}^{S^2-1}\sum_{j=0}^{B-1}
\mathbb{1}_{ij}^{\mathrm{obj}}
\left[(x_{ij}-\hat{x}_{ij})^2+(y_{ij}-\hat{y}_{ij})^2\right]
$$

여기서 hat은 정답을 뜻한다고 약속한다. 논문이나 코드마다 hat의 방향이 반대일 수 있으므로 식 앞에서 정의해야 한다.

### 7.2 너비·높이 손실

$$
L_{wh}=\lambda_{\mathrm{coord}}
\sum_{i=0}^{S^2-1}\sum_{j=0}^{B-1}
\mathbb{1}_{ij}^{\mathrm{obj}}
\left[\left(\sqrt{w_{ij}}-\sqrt{\hat{w}_{ij}}\right)^2
+\left(\sqrt{h_{ij}}-\sqrt{\hat{h}_{ij}}\right)^2\right]
$$

제곱근은 절대 크기가 같은 오차라도 작은 box에서 상대적으로 더 크게 반영한다. 예를 들어 너비가 $0.10\to0.20$인 변화와 $0.70\to0.80$인 변화는 원래 공간에서 모두 0.10이지만 제곱근 공간에서는 전자가 더 멀다.

예측 $w,h$가 음수면 제곱근이 `NaN`이 된다. 원형 구현을 재현할 때와 안정적 교육 구현을 만들 때를 구분해야 한다. 후자에서는 양수가 되도록 sigmoid, exponential, softplus 같은 매개화를 쓰고 clamp에 필요한 최소값을 둔다.

### 7.3 객체 confidence 손실

$$
L_{\mathrm{obj}}=
\sum_{i=0}^{S^2-1}\sum_{j=0}^{B-1}
\mathbb{1}_{ij}^{\mathrm{obj}}(q_{ij}-\hat{q}_{ij})^2
$$

### 7.4 비책임 confidence 손실

$$
L_{\mathrm{noobj}}=\lambda_{\mathrm{noobj}}
\sum_{i=0}^{S^2-1}\sum_{j=0}^{B-1}
\mathbb{1}_{ij}^{\mathrm{noobj}}(q_{ij}-0)^2
$$

객체가 있는 셀의 선택되지 않은 predictor도 `noobj` 항에 들어갈 수 있다. `noobj`를 단순히 "배경 셀"과 같은 말로 구현하면 mask가 달라진다.

### 7.5 class 손실

$$
L_{\mathrm{cls}}=
\sum_{i=0}^{S^2-1}\mathbb{1}_{i}^{\mathrm{obj}}
\sum_{c=1}^{C}(p_i(c)-\hat{p}_i(c))^2
$$

class 손실은 box predictor마다가 아니라 객체 중심이 든 셀마다 한 번 계산한다.

### 7.6 전체 손실

$$
L=L_{xy}+L_{wh}+L_{\mathrm{obj}}+L_{\mathrm{noobj}}+L_{\mathrm{cls}}
$$

원 논문의 대표 가중치는 $\lambda_{\mathrm{coord}}=5$, $\lambda_{\mathrm{noobj}}=0.5$다. 이는 모든 데이터셋의 정답이 아니라 당시 모델과 데이터의 선택이다.

## 8. PyTorch 최소 구현

다음 코드는 **실행 가능한 교육용 구현**이다. YOLOv1 논문의 fully connected head를 그대로 재현하지 않고 `1x1 Conv` head를 사용하며, 안정성을 위해 좌표와 크기에 sigmoid를 적용한다. 따라서 "YOLOv1의 책임·손실 구조를 shape 수준에서 검증하는 모델"이지 논문 수치 재현 코드가 아니다.

```python
import torch
from torch import nn
from torch.nn import functional as F


def split_prediction(raw, boxes_per_cell, num_classes):
    n, s1, s2, channels = raw.shape
    expected = 5 * boxes_per_cell + num_classes
    if s1 != s2 or channels != expected:
        raise ValueError(f"expected N x S x S x {expected}, got {tuple(raw.shape)}")
    box_raw = raw[..., : 5 * boxes_per_cell]
    box_raw = box_raw.reshape(n, s1, s2, boxes_per_cell, 5)
    class_raw = raw[..., 5 * boxes_per_cell :]
    return box_raw, class_raw


def cxcywh_to_xyxy(box):
    center, size = box[..., :2], box[..., 2:]
    return torch.cat((center - size / 2.0, center + size / 2.0), dim=-1)


def box_iou_aligned(a, b, eps=1e-9):
    left_top = torch.maximum(a[..., :2], b[..., :2])
    right_bottom = torch.minimum(a[..., 2:], b[..., 2:])
    intersection = (right_bottom - left_top).clamp_min(0).prod(dim=-1)
    area_a = (a[..., 2:] - a[..., :2]).clamp_min(0).prod(dim=-1)
    area_b = (b[..., 2:] - b[..., :2]).clamp_min(0).prod(dim=-1)
    union = area_a + area_b - intersection
    return intersection / union.clamp_min(eps)


def decode_boxes(box_raw):
    n, s, _, b, _ = box_raw.shape
    device, dtype = box_raw.device, box_raw.dtype
    rows, cols = torch.meshgrid(
        torch.arange(s, device=device, dtype=dtype),
        torch.arange(s, device=device, dtype=dtype),
        indexing="ij",
    )
    grid = torch.stack((cols, rows), dim=-1).view(1, s, s, 1, 2)
    xy = (box_raw[..., :2].sigmoid() + grid) / s
    wh = box_raw[..., 2:4].sigmoid()
    obj = box_raw[..., 4].sigmoid()
    return torch.cat((xy, wh), dim=-1), obj


def educational_yolo_loss(raw, target_box, target_class, object_mask, b, c):
    box_raw, class_raw = split_prediction(raw, b, c)
    pred_box, pred_obj = decode_boxes(box_raw)

    expanded_target = target_box.unsqueeze(-2).expand_as(pred_box)
    iou = box_iou_aligned(
        cxcywh_to_xyxy(pred_box),
        cxcywh_to_xyxy(expanded_target),
    ).detach()
    best = iou.argmax(dim=-1)
    responsible = F.one_hot(best, num_classes=b).to(raw.dtype)
    responsible = responsible * object_mask.unsqueeze(-1)

    n, s, _, _, _ = pred_box.shape
    rows, cols = torch.meshgrid(
        torch.arange(s, device=raw.device, dtype=raw.dtype),
        torch.arange(s, device=raw.device, dtype=raw.dtype),
        indexing="ij",
    )
    grid = torch.stack((cols, rows), dim=-1).view(1, s, s, 2)
    target_xy_cell = target_box[..., :2] * s - grid

    pred_xy_cell = box_raw[..., :2].sigmoid()
    xy_error = (pred_xy_cell - target_xy_cell.unsqueeze(-2)).square().sum(-1)
    pred_wh = pred_box[..., 2:4].clamp_min(1e-9)
    target_wh = target_box[..., 2:4].unsqueeze(-2).clamp_min(1e-9)
    wh_error = (pred_wh.sqrt() - target_wh.sqrt()).square().sum(-1)

    coord = 5.0 * ((xy_error + wh_error) * responsible).sum()
    object_loss = ((pred_obj - iou).square() * responsible).sum()
    no_object_loss = 0.5 * (pred_obj.square() * (1.0 - responsible)).sum()

    class_prob = class_raw.softmax(dim=-1)
    class_target = F.one_hot(target_class, num_classes=c).to(raw.dtype)
    class_error = (class_prob - class_target).square().sum(-1)
    class_loss = (class_error * object_mask).sum()
    return coord + object_loss + no_object_loss + class_loss


class TinyGridDetector(nn.Module):
    def __init__(self, boxes_per_cell=2, num_classes=3):
        super().__init__()
        self.b = boxes_per_cell
        self.c = num_classes
        self.backbone = nn.Sequential(
            nn.Conv2d(3, 16, 3, stride=2, padding=1),
            nn.ReLU(),
            nn.Conv2d(16, 32, 3, stride=2, padding=1),
            nn.ReLU(),
            nn.AdaptiveAvgPool2d((7, 7)),
        )
        self.head = nn.Conv2d(32, 5 * boxes_per_cell + num_classes, 1)

    def forward(self, image):
        nchw = self.head(self.backbone(image))
        return nchw.permute(0, 2, 3, 1).contiguous()


torch.manual_seed(7)
model = TinyGridDetector()
images = torch.randn(2, 3, 64, 64)
raw = model(images)
assert raw.shape == (2, 7, 7, 13)

target_box = torch.zeros(2, 7, 7, 4)
target_class = torch.zeros(2, 7, 7, dtype=torch.long)
object_mask = torch.zeros(2, 7, 7)
target_box[0, 2, 3] = torch.tensor([3.25 / 7, 2.50 / 7, 0.20, 0.40])
target_class[0, 2, 3] = 1
object_mask[0, 2, 3] = 1.0

loss = educational_yolo_loss(raw, target_box, target_class, object_mask, b=2, c=3)
assert torch.isfinite(loss)
loss.backward()
assert model.head.weight.grad is not None
assert torch.isfinite(model.head.weight.grad).all()
print("torch check passed:", tuple(raw.shape), f"loss={loss.item():.6f}")
```

### 구현에서 의도적으로 남긴 경계

- 이 target tensor는 셀마다 객체 하나만 표현한다. 이것이 YOLOv1의 구조적 한계를 그대로 드러낸다.
- `no_object_loss`는 모든 비책임 predictor를 포함한다. 큰 영상과 큰 $S$에서는 reduction과 batch 정규화를 별도로 설계해야 한다.
- 원 논문은 합제곱 기반이지만 최신 구현은 BCE, IoU 계열 box loss, objectness 분리 등 다른 선택을 쓴다.
- 학습 target을 만드는 데이터 파이프라인에서는 동일 셀 충돌 정책을 명시해야 한다. 조용히 마지막 label로 덮어쓰면 데이터 손실이 보이지 않는다.

## 9. 후처리: threshold와 NMS

head가 만든 $S^2B$개 box에 대해 class score를 계산한다. VOC 설정에서는 $7^2\times2=98$개라 작지만, 더 촘촘한 detector에서는 후보가 훨씬 많다.

class별 NMS의 기본 절차는 다음과 같다.

1. class score가 threshold보다 낮은 후보를 제거한다.
2. 남은 후보를 score 내림차순으로 정렬한다.
3. 가장 높은 box를 보존한다.
4. 그 box와 IoU가 threshold보다 큰 같은 class box를 억제한다.
5. 후보가 빌 때까지 반복한다.

NMS는 학습 손실이 아니라 추론 후처리다. 서로 다른 class에 class별 NMS를 적용하면 같은 위치의 box가 여러 class로 남을 수 있다. class-agnostic NMS는 이를 줄이지만 서로 겹친 실제 다른 클래스 객체를 지울 수 있다.

### threshold의 순서

`objectness`만 먼저 자르면 class score가 높은 희귀 후보를 잃을 수 있고, 모든 class score를 펼친 뒤 자르면 후보 수와 메모리가 늘어난다. 출력 계약, latency budget, 데이터의 class 수에 맞춰 순서를 정하고 정확도 회귀 테스트를 둔다.

## 10. C++ 좌표 디코더

다음 코드는 외부 라이브러리가 필요 없는 **실행 가능한 C++17 예제**다. 네트워크 추론은 포함하지 않고, 모델 출력 이후의 좌표 계약만 검증한다.

```cpp
#include <array>
#include <cassert>
#include <cmath>
#include <iomanip>
#include <iostream>

using Box = std::array<double, 4>;

Box decode_cell(int row, int col, const Box& cell_box, int grid_size) {
    const double tx = cell_box[0];
    const double ty = cell_box[1];
    const double width = cell_box[2];
    const double height = cell_box[3];
    const double cx = (static_cast<double>(col) + tx) / grid_size;
    const double cy = (static_cast<double>(row) + ty) / grid_size;
    return {
        cx - width / 2.0,
        cy - height / 2.0,
        cx + width / 2.0,
        cy + height / 2.0,
    };
}

int main() {
    const Box actual = decode_cell(2, 3, {0.25, 0.50, 0.20, 0.40}, 7);
    const Box expected = {0.3642857143, 0.1571428571, 0.5642857143, 0.5571428571};
    for (std::size_t i = 0; i < actual.size(); ++i) {
        assert(std::abs(actual[i] - expected[i]) < 1e-9);
    }
    std::cout << std::fixed << std::setprecision(7);
    for (double value : actual) {
        std::cout << value << ' ';
    }
    std::cout << "\nC++ decode passed\n";
```

## 11. C# 좌표 디코더

다음 코드는 **실행 가능한 C# 예제**이며 C++ 및 NumPy와 같은 golden value를 사용한다.

```csharp
using System;

public static class YoloDecode
{
    public static double[] DecodeCell(int row, int col, double[] cellBox, int gridSize)
    {
        double tx = cellBox[0];
        double ty = cellBox[1];
        double width = cellBox[2];
        double height = cellBox[3];
        double cx = (col + tx) / gridSize;
        double cy = (row + ty) / gridSize;
        return new double[] {
            cx - width / 2.0,
            cy - height / 2.0,
            cx + width / 2.0,
            cy + height / 2.0
        };
    }

    public static void Main()
    {
        double[] actual = DecodeCell(2, 3, new double[] { 0.25, 0.50, 0.20, 0.40 }, 7);
        double[] expected = new double[] {
            0.3642857143, 0.1571428571, 0.5642857143, 0.5571428571
        };
        for (int i = 0; i < actual.Length; ++i) {
            if (Math.Abs(actual[i] - expected[i]) >= 1e-9) {
                throw new Exception("decode mismatch at index " + i);
            }
        }
        Console.WriteLine(String.Join(" ", Array.ConvertAll(actual, x => x.ToString("F7"))));
        Console.WriteLine("C# decode passed");
    }
}
```

## 12. 프레임워크 간 shape·layout·dtype 계약

| 계층 | PyTorch Python | C++ 추론 런타임 | C# 추론 런타임 | 확인할 계약 |
| --- | --- | --- | --- | --- |
| 입력 | 보통 `NCHW`, `float32` | 엔진별 `NCHW` 또는 `NHWC` | API별 상이 | resize, channel 순서, 정규화 |
| head 원출력 | `N x D x S x S` 가능 | export graph에 고정 | export graph에 고정 | $D=5B+C$ |
| 논리 파싱 뷰 | `N x S x S x D` | stride 기반 접근 | 다차원 배열 또는 flat buffer | 단순 reshape와 transpose 구분 |
| box | 정규화 `float32` | `float32` 권장 | `float` 또는 `double` | `cxcywh`와 `xyxy` 구분 |
| score | `float32` | FP16 계산 후 FP32 NMS 가능 | 보통 `float` | activation 포함 여부 |
| index | `torch.long` | `int32`/`int64` | `int`/`long` | class id와 batch id |

ONNX로 내보낼 때 출력 이름을 `predictions` 하나로 두는 것만으로는 부족하다. 모델 카드나 sidecar schema에 다음을 적어야 한다.

```text
layout: N x S x S x (5B+C)
box_format: normalized_cxcywh
box_order: B blocks of [tx, ty, w, h, objectness], then C class scores
activations: sigmoid(xywh,obj), softmax(class) applied outside graph
grid_size: 7
```

축 약어만 나열하면 오해하기 쉽다. 실제 계약은 위 예시처럼 각 축을 구분해 쓰고, 가능하면 정적·동적 여부도 기록한다.

## 13. 테스트 전략

### 13.1 최소 단위 테스트

- 출력 마지막 축이 정확히 $5B+C$인지 검사한다.
- `NCHW -> NHWC`가 `permute`로 수행되는지 검사한다.
- 셀 $(0,0)$의 offset $(0,0)$과 셀 $(S-1,S-1)$의 offset $(1,1)$ 경계를 검사한다.
- `cxcywh -> xyxy -> cxcywh` 왕복 오차를 검사한다.
- 동일 box의 IoU가 1인지 검사한다.
- 겹치지 않는 box의 IoU가 0인지 검사한다.
- 0 넓이 box에서 `NaN`이 나오지 않는지 검사한다.
- 한 셀의 두 predictor 중 더 높은 IoU를 가진 하나만 책임지는지 검사한다.
- 객체가 없는 batch에서도 loss와 gradient가 유한한지 검사한다.
- NumPy, C++, C# 디코더가 같은 golden value를 내는지 검사한다.

### 13.3 통합 테스트

작은 고정 seed batch 하나를 저장하고 다음을 함께 비교한다.

1. raw tensor의 shape와 checksum
2. 디코딩 후 첫 후보의 좌표
3. threshold 이후 후보 수
4. NMS 이후 `(class_id, score, box)`

부동소수점 backend가 달라 bitwise 일치가 어렵다면 좌표·score별 허용 오차를 명시한다. 후보 순서가 score tie에 따라 달라질 수 있으므로 안정 정렬 기준도 정한다.

## 14. 디버깅 지도

| 증상 | 가능한 원인 | 먼저 할 검사 |
| --- | --- | --- |
| box가 대각선 방향으로 반복 이동 | row와 column, $x$와 $y$를 뒤바꿈 | 단일 셀 golden decode |
| 너비와 높이는 맞지만 중심이 한 셀씩 밀림 | 0-based/1-based grid 혼용 | $(c+t_x)/S$ 식과 index 범위 |
| 모든 box가 화면 왼쪽 위에 몰림 | grid offset을 더하지 않음 | decoded center의 min/max |
| class가 box마다 엉뚱하게 반복 | $C$ 채널을 predictor마다 있다고 파싱 | 마지막 축 slice 표 |
| loss가 갑자기 `NaN` | 음수 $w,h$에 제곱근, 0 union, FP16 overflow | 첫 non-finite tensor hook |
| 배경에서 objectness가 높음 | no-object mask 또는 가중치 오류 | 객체 없는 synthetic batch |
| 작은 객체 recall이 낮음 | 같은 셀 충돌, 거친 $7\times7$ grid | 중심 셀 occupancy histogram |
| Python과 배포 결과가 다름 | layout, activation, resize, color 순서 차이 | export 전후 raw output 비교 |

gradient가 유한한지만 보는 것으로는 부족하다. 책임 predictor가 매 step 급격히 바뀌면 loss 곡선에 불연속적인 흔들림이 생길 수 있다. 셀별 `best predictor` 비율과 평균 IoU도 함께 기록하면 assignment 이상을 빨리 찾는다.

## 15. 성능·메모리·수치 안정성

### 15.1 출력 메모리

출력 원소 수는 다음과 같다.

$$
M=N S^2(5B+C)
$$

`float32` byte 수는 $4M$이다. YOLOv1 VOC에서 $N=32$이면 다음과 같다.

$$
32\times7^2\times30\times4=188{,}160\ \mathrm{bytes}
$$

최종 출력 자체는 작다. 실제 메모리의 대부분은 backbone activation, gradient, optimizer state가 차지한다. 따라서 출력 tensor만 보고 전체 학습 batch 크기를 결정하면 안 된다.

### 15.2 `permute`와 contiguous copy

`permute`는 대개 stride만 바꾸는 view지만, 뒤에서 `.contiguous()`를 호출하면 복사가 생긴다. 손실 구현이 `NCHW`를 직접 파싱할 수 있다면 copy를 피할 수 있다. 다만 축 계약이 복잡해져 버그 위험이 커질 수 있으므로 profiler 결과를 보고 선택한다.

### 15.3 mixed precision

- IoU의 작은 union은 FP16에서 underflow 위험이 있다.
- NMS와 누적 loss는 FP32로 올리는 편이 안전할 수 있다.
- 제곱근 전에는 최소 양수 clamp가 필요하다.
- score threshold 근처에서는 FP16 반올림 때문에 후보 수가 달라질 수 있다.
- 안정성을 고쳤다는 이유로 모든 activation을 무조건 FP32로 두면 latency와 memory 이점을 잃는다. 위험 연산만 선택적으로 승격한다.

### 15.4 latency는 평균만 보지 않는다

실시간 시스템은 평균 FPS보다 p95/p99 latency와 queueing이 중요하다. preprocessing, inference, decode, NMS, device copy를 따로 측정한다. 후보가 많은 장면에서 NMS 시간이 튀는지 확인하고 최대 후보 수 제한의 정확도 영향을 함께 기록한다.

## 16. 실무 실패 사례

### 사례 A: resize 방식 불일치

학습은 이미지를 강제로 정사각형으로 늘렸는데 배포는 aspect ratio를 보존한 letterbox를 썼다. 모델 좌표는 맞아도 원본 영상으로 되돌리는 역변환이 달라 box가 세로로 밀렸다.

대응은 전처리 변환을 affine matrix로 기록하고, 후처리가 그 역행렬을 사용하게 하는 것이다. 단순히 `x *= original_width`만 해서는 padding을 제거할 수 없다.

### 사례 B: 같은 셀의 여러 객체가 조용히 사라짐

target builder가 `(row, col)` key 하나에 label 하나만 저장했다. 군중 장면에서 나중 객체가 이전 객체를 덮어썼지만 에러가 없었다.

대응은 cell collision count를 metric으로 남기고, 충돌 정책을 명시하는 것이다. YOLOv1을 유지한다면 한 객체를 선택해야 하지만, 데이터 손실 규모를 알아야 다음 구조를 선택할 수 있다.

### 사례 C: activation을 두 번 적용

export graph 안에 sigmoid가 포함됐는데 C# 후처리도 sigmoid를 다시 적용했다. objectness가 0.5 근처로 눌리고 threshold tuning이 무너졌다.

대응은 raw/logit 여부를 출력 schema에 기록하고, 알려진 logit에 대한 golden output을 언어별 테스트로 공유하는 것이다.

### 사례 D: class별 NMS가 아니라 전체 NMS를 적용

사람이 자전거 앞에 있는 장면에서 두 box가 많이 겹쳤다. class-agnostic NMS가 둘 중 하나를 지워 recall이 떨어졌다.

대응은 제품 요구에 맞는 NMS 정책을 고르고, 겹치는 서로 다른 클래스의 regression case를 유지하는 것이다.

## 17. 배포 관점

배포 전에 다음 artifact를 모델과 함께 versioning한다.

- 입력 크기, 색 순서, normalization, resize 및 padding 규칙
- 출력 tensor 이름, shape, layout, dtype
- box 좌표 형식과 정규화 기준
- graph 안팎의 activation 경계
- score 계산식과 threshold
- NMS의 class별 여부, IoU threshold, 최대 후보 수
- class index와 label 문자열의 대응
- 학습 코드 commit, export 도구 버전, opset
- golden input과 raw/decode/NMS 단계별 expected output

모니터링에서는 단순 요청 성공률 외에 다음 분포를 본다.

- 이미지당 탐지 개수
- class별 score와 box 크기
- 화면 경계에 닿는 box 비율
- NMS 전후 후보 수
- 빈 결과 비율
- preprocessing, inference, postprocessing latency
- 입력 해상도와 aspect ratio drift

label이 늦게 도착하는 시스템이라도 이 분포들은 파이프라인 파손을 조기에 알려준다. 다만 분포 변화가 곧 정확도 하락이라는 뜻은 아니므로 ground-truth 기반 평가를 완전히 대체하지는 못한다.

## 19. 연습문제

### 문제 1

$S=7$, $B=2$, $C=20$, batch 16일 때 논리 출력 shape와 `float32` byte 수를 구하라.

### 문제 2

객체 중심이 $(x_c,y_c)=(0.52,0.10)$일 때 0-based 책임 셀 $(r,c)$를 구하라. 단, $r=\lfloor Sy_c\rfloor$, $c=\lfloor Sx_c\rfloor$를 사용한다.

### 문제 3

셀 $(r,c)=(4,1)$에서 $(t_x,t_y,w,h)=(0.4,0.2,0.3,0.2)$를 예측했다. 정규화 `cxcywh`와 `xyxy`를 구하라.

### 문제 4

한 셀의 predictor 두 개가 정답과 각각 IoU 0.3, 0.7을 갖는다. 어떤 predictor가 좌표 손실을 받는가? 다른 predictor는 완전히 손실에서 제외되는가?

### 문제 5

class 확률 $p_{\mathrm{cat}}=0.8$, confidence $q=0.6$이면 cat score는 얼마인가? 이것을 반드시 "고양이일 확률 48%"라고 말해도 되는가?

### 문제 6

Python에서는 올바른 box가 나오지만 C#에서 모든 중심이 왼쪽 위로 몰렸다. 가장 먼저 비교할 중간값 세 가지를 제안하라.

## 20. 해답

### 해답 1

$5B+C=30$이므로 shape는 `16 x 7 x 7 x 30`이다. 원소 수와 byte 수는 다음과 같다.

$$
16\times7\times7\times30=23{,}520
$$

$$
23{,}520\times4=94{,}080\ \mathrm{bytes}
$$

### 해답 2

$$
r=\lfloor7\times0.10\rfloor=0,\qquad
c=\lfloor7\times0.52\rfloor=3
$$

따라서 책임 셀은 $(0,3)$이다.

### 해답 3

$$
x_c=\frac{1+0.4}{7}=0.2,\qquad
y_c=\frac{4+0.2}{7}=0.6
$$

따라서 `cxcywh`는 $(0.2,0.6,0.3,0.2)$이고 `xyxy`는 다음과 같다.

$$
(0.05,0.50,0.35,0.70)
$$

### 해답 4

IoU 0.7인 predictor가 책임을 맡아 좌표, 크기, object confidence 손실을 받는다. 다른 predictor는 좌표·크기 손실을 받지 않지만 비책임 confidence 손실을 받을 수 있으므로 완전히 제외되는 것은 아니다.

### 해답 5

$$
s_{\mathrm{cat}}=0.8\times0.6=0.48
$$

이는 ranking에 쓰는 class-specific detection score다. calibration을 검증하지 않았다면 현실의 사건 확률 48%라고 단정하지 않는다.

### 해답 6

첫째 raw output에서 해당 셀의 `tx,ty`; 둘째 grid offset을 더한 `c+tx,r+ty`; 셋째 $S$로 나눈 정규화 중심 $x_c,y_c$를 비교한다. 그다음 layout transpose, sigmoid 적용 위치, row/column 순서를 확인한다.

## 핵심 요약

- YOLOv1은 검출을 $S\times S$ 격자의 고정 크기 회귀로 바꿨다.
- 출력은 $N\times S\times S\times(5B+C)$이며 class 분포는 box마다가 아니라 셀마다 한 벌이다.
- 객체 중심이 책임 셀을 정하고, 그 셀의 predictor 중 정답과 IoU가 가장 큰 하나가 box를 책임진다.
- 셀 offset은 $(c+t_x)/S$, $(r+t_y)/S$로 이미지 좌표가 된다.
- 전체 손실은 중심, 크기, 객체 confidence, 비책임 confidence, class의 다섯 항으로 읽을 수 있다.
- 제곱근 크기 손실과 IoU 분모에는 수치 안정성 처리가 필요하다.
- class score는 조건부 class 확률과 confidence의 곱이며 NMS 이전 후보 ranking에 쓰인다.
- Python, C++, C#의 차이보다 layout, activation, 좌표 형식 계약의 차이가 더 큰 장애 원인이다.
- YOLOv1의 거친 grid와 셀당 class 한 벌은 같은 셀의 여러 작은 객체에서 구조적 한계를 만든다.

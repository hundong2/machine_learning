# CNN의 출발점: 2D 합성곱, 텐서 크기, 수용영역

- 날짜: 2026-07-18
- 예상 학습 시간: 90~120분
- 난이도: 입문 → 초급 실무
- 연결 원본: `02.ClassificationForHands.md`, `02-1.LazyLinear.md`, `02-02.AdaptiveAvgPooll2d.md`, `02-03.SPP.md`
- 실습 기준: Python 3.12.12, PyTorch 2.13.0, torchvision 0.28.0, C++17, TorchSharp-cpu 0.107.0

오늘은 손동작 이미지 분류 코드에서 곧바로 `Conv2d`를 사용하는 대신, 합성곱 층이 실제로 무엇을 계산하고 텐서 크기가 왜 변하는지부터 살펴본다. 이 내용을 알아야 다음 주제인 Global Average Pooling, Adaptive Pooling, SPP, Fast R-CNN에서 발생하는 shape 문제를 암기가 아니라 계산으로 해결할 수 있다.

## 1. 학습 목표

학습을 마치면 다음을 할 수 있어야 한다.

1. 딥러닝 프레임워크의 `Conv2d`가 수학적 합성곱이 아닌 **교차상관(cross-correlation)** 을 계산한다는 사실을 설명한다.
2. 입력과 kernel, stride, padding, dilation으로 출력의 높이와 너비를 계산한다.
3. 일반 합성곱과 group/depthwise 합성곱의 파라미터 수와 MAC 수를 계산한다.
4. 여러 층을 쌓았을 때 수용영역(receptive field)을 계산한다.
5. PyTorch로 수동 교차상관과 `nn.Conv2d`의 결과가 같은지 테스트한다.
6. Python, C++, C# 구현 사이의 tensor layout과 메모리 관리 차이를 구분한다.
7. shape, dtype, channel order, 재현성 때문에 생기는 실무 장애를 예방한다.

## 2. 선수 지식

### 2.1 이미지도 결국 숫자 배열이다

RGB 이미지 한 장은 보통 높이 $H$, 너비 $W$, 채널 $C=3$인 배열이다. 라이브러리마다 저장 순서가 다르다.

| 환경 | 대표 layout | RGB/BGR | 배치가 있을 때 |
| --- | --- | --- | --- |
| OpenCV | HWC | 기본 BGR | 직접 관리 |
| NumPy/Albumentations | HWC | 보통 RGB로 변환 | NHWC 가능 |
| PyTorch/TorchSharp | CHW | 코드가 정함 | NCHW |

PyTorch의 일반적인 이미지 배치는 다음 shape을 갖는다.

$$
X \in \mathbb{R}^{N \times C_{in} \times H_{in} \times W_{in}}
$$

- $N$: batch 크기
- $C_{in}$: 입력 채널 수
- $H_{in}, W_{in}$: 입력 높이와 너비

`HWC` 데이터를 바로 `Conv2d`에 넣으면 채널과 높이를 혼동한다. OpenCV로 읽은 이미지는 BGR이므로 RGB 학습 파이프라인이라면 색 순서까지 바꿔야 한다.

### 2.2 CNN이 선형층보다 이미지에 잘 맞는 이유

$64 \times 64$ RGB 이미지를 128개 뉴런과 완전 연결하면 첫 층에만 대략

$$
64 \times 64 \times 3 \times 128 \approx 1.57\text{M}
$$

개의 가중치가 필요하다. 반면 $3 \times 3$ kernel 16개를 사용하는 합성곱은

$$
16 \times 3 \times 3 \times 3 = 432
$$

개의 kernel 가중치만 필요하다. 같은 kernel을 모든 위치에 재사용하는 **가중치 공유(weight sharing)** 와 가까운 픽셀만 보는 **지역 연결(local connectivity)** 덕분이다.

## 3. 합성곱의 직관: 작은 탐지기를 미끄러뜨린다

kernel은 작은 패턴 탐지기다. 예를 들어 다음 kernel은 오른쪽 아래 방향의 밝기 차이에 반응한다.

$$
K =
\begin{bmatrix}
1 & 0 \\
0 & -1
\end{bmatrix}
$$

kernel을 이미지의 각 위치에 겹치고, 대응하는 원소끼리 곱한 뒤 더한다. 이 연산은 모서리, 선, 질감 같은 국소 패턴의 존재를 하나의 값으로 요약한다. 학습 CNN에서는 사람이 kernel 값을 정하지 않는다. 손실 함수의 gradient를 따라 kernel이 데이터에 유용한 패턴을 탐지하도록 바뀐다.

### 3.1 합성곱과 교차상관은 다르다

엄밀한 수학적 합성곱은 kernel을 상하·좌우로 뒤집은 뒤 계산한다. 교차상관은 뒤집지 않는다. PyTorch `Conv2d`와 OpenCV `filter2D`는 실제로 교차상관을 계산한다.

학습 가능한 CNN에서는 큰 문제가 되지 않는다. 원하는 뒤집힌 kernel도 학습 과정에서 그대로 배울 수 있기 때문이다. 그러나 수동 구현 결과를 비교하거나 사전에 설계한 영상 처리 kernel을 옮길 때는 차이를 알아야 한다.

## 4. 다중 채널 2D 교차상관의 수학

가장 일반적인 `groups=1`을 생각하자. 가중치와 bias의 shape은 다음과 같다.

$$
W \in \mathbb{R}^{C_{out} \times C_{in} \times K_h \times K_w},
\qquad
b \in \mathbb{R}^{C_{out}}
$$

출력 $Y \in \mathbb{R}^{N \times C_{out} \times H_{out} \times W_{out}}$의 한 원소는

$$
Y_{n,o,i,j}
= b_o
+ \sum_{c=0}^{C_{in}-1}
  \sum_{u=0}^{K_h-1}
  \sum_{v=0}^{K_w-1}
W_{o,c,u,v}\,
\widetilde{X}_{n,c,\,iS_h+uD_h,\,jS_w+vD_w}
$$

이다.

- $o$: 출력 채널 인덱스
- $S_h,S_w$: stride
- $D_h,D_w$: dilation
- $\widetilde{X}$: padding이 적용된 입력

중요한 점은 출력 채널 하나가 기본적으로 **모든 입력 채널의 결과를 합산**한다는 것이다. RGB 이미지에 $3 \times 3$ kernel 하나를 쓴다는 말은 실제로 $3 \times 3 \times 3$개의 가중치를 쓴다는 뜻이다.

## 5. 출력 크기를 계산하는 공식

dilation을 적용한 kernel의 유효 크기는

$$
K_{eff} = D(K-1)+1
$$

이다. 한 공간 축의 출력 크기는

$$
O = \left\lfloor
\frac{I + 2P - D(K-1) - 1}{S} + 1
\right\rfloor
=
\left\lfloor
\frac{I + 2P - K_{eff}}{S} + 1
\right\rfloor
$$

로 계산한다. 높이와 너비에 각각 적용하면 된다.

### 5.1 손으로 계산하기

입력이 `(N, 3, 64, 64)`이고 다음 층을 사용한다고 하자.

```python
nn.Conv2d(3, 16, kernel_size=3, stride=2, padding=1, dilation=1)
```

높이는

$$
H_{out}
= \left\lfloor
\frac{64 + 2(1) - 1(3-1) - 1}{2} + 1
\right\rfloor
= 32
$$

이다. 따라서 출력은 `(N, 16, 32, 32)`다.

### 5.2 `same`이라는 표현을 조심하자

홀수 kernel, stride 1, dilation 1일 때 $P=(K-1)/2$로 두면 공간 크기가 유지된다. 하지만 stride가 2이면 출력 크기가 줄어들며, 라이브러리마다 `same` padding의 세부 규칙이 다를 수 있다. PyTorch의 `padding="same"`은 현재 stride 1만 지원하므로 운영 코드는 암묵적 기대보다 명시적 padding과 shape 테스트가 안전하다.

## 6. 파라미터 수와 연산량

group 수를 $G$라고 하면 가중치 파라미터 수는

$$
C_{out}\frac{C_{in}}{G}K_hK_w
$$

이다. bias를 사용하면 $C_{out}$을 더한다.

출력 한 원소를 계산할 때 필요한 곱셈-누산(MAC) 수는 $\frac{C_{in}}{G}K_hK_w$이므로, 한 batch의 대략적인 MAC 수는

$$
N H_{out}W_{out}C_{out}\frac{C_{in}}{G}K_hK_w
$$

이다. 한 MAC을 곱셈 1회와 덧셈 1회로 세는 도구는 FLOPs를 약 $2\times\text{MACs}$로 보고한다. 프로파일러마다 정의가 다르므로 숫자를 비교할 때 기준을 확인해야 한다.

예를 들어 `Conv2d(3, 16, 3, padding=1)`에 $64 \times 64$ 이미지 한 장을 넣으면:

- bias 포함 파라미터: $16(3\times3\times3+1)=448$
- MACs: $64\times64\times16\times3\times3\times3=1{,}769{,}472$

파라미터 수가 적다고 실행이 항상 빠른 것은 아니다. 큰 feature map에서는 같은 kernel이 매우 많은 위치에서 실행되므로 activation 메모리와 MAC가 커진다.

## 7. Group convolution과 depthwise convolution

`groups=G`는 입력·출력 채널 연결을 $G$개 그룹으로 나눈다. 입력 채널과 출력 채널은 모두 $G$로 나누어져야 한다.

- `groups=1`: 모든 입력 채널과 출력 채널이 연결된다.
- `groups=C_in`이고 `C_out=K C_in`: 각 입력 채널을 독립적으로 처리하는 depthwise convolution이다.
- depthwise 뒤에 $1\times1$ pointwise convolution을 붙이면 MobileNet류의 depthwise separable convolution이 된다.

그룹을 늘리면 파라미터와 MAC는 대략 $1/G$로 줄지만, 채널 사이 정보 교환도 줄어든다. 연산량 감소가 하드웨어 실측 지연 감소로 그대로 이어지지 않을 수 있으므로 배포 장치에서 benchmark해야 한다.

## 8. 수용영역: 출력 한 점이 원본의 어디까지 보는가

수용영역은 feature map의 한 원소에 영향을 줄 수 있는 입력 영역이다. 층 $l$의 kernel 유효 크기를 $K_l^{eff}$, stride를 $S_l$이라고 하자. 이전 층 기준 이동 간격(jump) $J_l$과 수용영역 $R_l$은 다음 점화식으로 계산한다.

$$
J_0=1,\quad R_0=1
$$

$$
J_l=J_{l-1}S_l
$$

$$
R_l=R_{l-1}+(K_l^{eff}-1)J_{l-1}
$$

예를 들어 `Conv3x3(s=1) → MaxPool2x2(s=2) → Conv3x3(s=1)`이면:

| 층 | $J$ | $R$ |
| --- | ---: | ---: |
| 입력 | 1 | 1 |
| Conv 3×3 | 1 | 3 |
| Pool 2×2 | 2 | 4 |
| Conv 3×3 | 2 | 8 |

마지막 feature 한 점은 이론적으로 입력의 $8\times8$ 영역을 본다. 객체 전체를 분류하려면 충분히 넓은 수용영역이 필요하지만, 지나친 downsampling은 작은 물체와 경계 정보를 잃게 한다. 이후 SPP와 객체 탐지에서 이 균형이 중요해진다.

## 9. Python 실습 1: 수동 교차상관과 `Conv2d` 비교

다음 코드는 외부 데이터 없이 그대로 실행할 수 있다.

<!-- test:python:start -->
```python
from __future__ import annotations

import torch
from torch import nn
from torch.nn import functional as F


def xcorr2d_valid(x: torch.Tensor, kernel: torch.Tensor) -> torch.Tensor:
    """단일 채널 NCHW 입력에 stride=1인 valid 교차상관을 수행한다."""
    if x.ndim != 4 or x.shape[1] != 1:
        raise ValueError("x must have shape (N, 1, H, W)")
    if kernel.ndim != 2:
        raise ValueError("kernel must have shape (Kh, Kw)")

    kh, kw = kernel.shape
    if x.shape[-2] < kh or x.shape[-1] < kw:
        raise ValueError("kernel cannot be larger than input")

    # (N, 1, Hout, Wout, Kh, Kw)
    windows = x.unfold(2, kh, 1).unfold(3, kw, 1)
    return (windows * kernel.view(1, 1, 1, 1, kh, kw)).sum((-1, -2))


torch.manual_seed(42)

x = torch.tensor(
    [[
        [1.0, 2.0, 0.0, 1.0],
        [3.0, 1.0, 2.0, 2.0],
        [0.0, 1.0, 3.0, 1.0],
        [2.0, 2.0, 1.0, 0.0],
    ]]
).unsqueeze(0)  # (N=1, C=1, H=4, W=4)

kernel = torch.tensor([[1.0, 0.0], [0.0, -1.0]])
expected = xcorr2d_valid(x, kernel)

conv = nn.Conv2d(1, 1, kernel_size=2, bias=False)
with torch.no_grad():
    conv.weight.copy_(kernel.view(1, 1, 2, 2))

actual = conv(x)
torch.testing.assert_close(actual, expected)
print(actual)
print("manual xcorr == nn.Conv2d:", torch.equal(actual, expected))


class TinyCNN(nn.Module):
    def __init__(self, num_classes: int = 11) -> None:
        super().__init__()
        self.features = nn.Sequential(
            nn.Conv2d(3, 16, kernel_size=3, padding=1, bias=False),
            nn.BatchNorm2d(16),
            nn.ReLU(inplace=True),
            nn.MaxPool2d(kernel_size=2),              # 64 -> 32
            nn.Conv2d(16, 32, kernel_size=3, padding=1, bias=False),
            nn.BatchNorm2d(32),
            nn.ReLU(inplace=True),
            nn.MaxPool2d(kernel_size=2),              # 32 -> 16
        )
        self.pool = nn.AdaptiveAvgPool2d((1, 1))
        self.classifier = nn.Linear(32, num_classes)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        x = self.features(x)
        x = self.pool(x)
        x = torch.flatten(x, start_dim=1)
        return self.classifier(x)


model = TinyCNN(num_classes=11)
batch = torch.randn(4, 3, 64, 64)
labels = torch.tensor([0, 1, 2, 3])

logits = model(batch)
assert logits.shape == (4, 11)

loss = F.cross_entropy(logits, labels)
loss.backward()
assert all(
    parameter.grad is None or torch.isfinite(parameter.grad).all()
    for parameter in model.parameters()
)

trainable = sum(p.numel() for p in model.parameters() if p.requires_grad)
print("logits shape:", tuple(logits.shape))
print("loss:", float(loss.detach()))
print("trainable parameters:", trainable)
```

### 9.1 왜 테스트를 코드에 넣는가

- `torch.testing.assert_close`: 수동 수식과 프레임워크 구현의 수치 일치를 확인한다.
- `assert logits.shape == (4, 11)`: 모델 구조 변경으로 classifier 입력이 깨지는 회귀를 잡는다.
- gradient 유한성 검사: `NaN`과 `Inf`가 첫 학습 단계부터 발생하는지 확인한다.

실무에서는 임의의 한 입력 크기만 검사하지 말고, 모델이 지원해야 하는 최소·대표·최대 해상도를 parameterized test로 검사한다.

## 10. Python 실습 2: shape 추적과 재현성

모델을 디버깅할 때 `print(x.shape)`를 여기저기 넣기보다 forward hook으로 경계를 관찰할 수 있다.

```python
def log_shape(name: str):
    def hook(module, inputs, output):
        input_shape = tuple(inputs[0].shape)
        output_shape = tuple(output.shape)
        print(f"{name:12s}: {input_shape} -> {output_shape}")
    return hook


handles = []
for name, module in model.named_modules():
    if isinstance(module, (nn.Conv2d, nn.MaxPool2d, nn.AdaptiveAvgPool2d, nn.Linear)):
        handles.append(module.register_forward_hook(log_shape(name)))

with torch.inference_mode():
    _ = model(torch.randn(1, 3, 64, 64))

for handle in handles:
    handle.remove()
```

재현성이 필요한 회귀 테스트에서는 다음 설정을 고려한다.

```python
import random
import numpy as np
import torch

seed = 42
random.seed(seed)
np.random.seed(seed)
torch.manual_seed(seed)

# 디버깅/회귀 테스트용. 일부 연산은 오류를 내거나 느려질 수 있다.
torch.use_deterministic_algorithms(True)
torch.backends.cudnn.benchmark = False
```

같은 seed가 모든 PyTorch 버전, 플랫폼, CPU/GPU 사이에서 bitwise 동일성을 보장하지는 않는다. 따라서 실험 기록에는 seed뿐 아니라 PyTorch/CUDA/cuDNN 버전, 장치, 데이터 split과 commit SHA도 남겨야 한다.

## 11. C++17 실습: 의존성 없이 valid 교차상관 구현

C++에서는 메모리 layout과 경계 검사를 직접 책임지는 경우가 많다. 다음 예제는 단일 채널 교차상관의 핵심 루프를 보여준다.

<!-- test:cpp:start -->
```cpp
#include <cmath>
#include <iostream>
#include <stdexcept>
#include <vector>

using Matrix = std::vector<std::vector<float>>;

Matrix xcorr_valid(const Matrix& x, const Matrix& kernel) {
    if (x.empty() || x.front().empty() ||
        kernel.empty() || kernel.front().empty()) {
        throw std::invalid_argument("input and kernel must not be empty");
    }

    const std::size_t h = x.size();
    const std::size_t w = x.front().size();
    const std::size_t kh = kernel.size();
    const std::size_t kw = kernel.front().size();
    if (h < kh || w < kw) {
        throw std::invalid_argument("kernel cannot be larger than input");
    }

    const std::size_t out_h = h - kh + 1;
    const std::size_t out_w = w - kw + 1;
    Matrix y(out_h, std::vector<float>(out_w, 0.0F));

    for (std::size_t i = 0; i < out_h; ++i) {
        for (std::size_t j = 0; j < out_w; ++j) {
            float sum = 0.0F;
            for (std::size_t u = 0; u < kh; ++u) {
                for (std::size_t v = 0; v < kw; ++v) {
                    sum += x[i + u][j + v] * kernel[u][v];
                }
            }
            y[i][j] = sum;
        }
    }
    return y;
}

int main() {
    const Matrix x{
        {1, 2, 0, 1},
        {3, 1, 2, 2},
        {0, 1, 3, 1},
        {2, 2, 1, 0},
    };
    const Matrix kernel{{1, 0}, {0, -1}};
    const Matrix y = xcorr_valid(x, kernel);

    if (y.size() != 3 || y.front().size() != 3) {
        throw std::runtime_error("unexpected output shape");
    }
    for (const auto& row : y) {
        for (float value : row) std::cout << value << ' ';
        std::cout << '\n';
    }
}
```

```bash
./xcorr
```

운영 코드에서는 이 루프 대신 OpenCV `cv::filter2D`, LibTorch `torch::nn::Conv2d`, oneDNN, CUDA/cuDNN 같은 최적화 구현을 쓴다. OpenCV의 `filter2D`는 출력 크기를 입력과 같게 만들기 위해 border를 처리하고 기본적으로 HWC/BGR 이미지를 다룬다는 점이 위의 valid/NCHW 예제와 다르다. kernel을 뒤집지 않는 교차상관이라는 점은 같다.

## 12. C# 실습: TorchSharp로 동일한 shape 계약 만들기

TorchSharp는 LibTorch를 .NET에서 사용하는 바인딩이다. 2026-07-18 기준 CPU 번들 패키지 예시는 다음과 같다.

```bash
dotnet new console -n TinyCnnCs -f net8.0
cd TinyCnnCs
dotnet add package TorchSharp-cpu --version 0.107.0
```

`Program.cs`:

```csharp
using System;
using System.Linq;
using TorchSharp;
using static TorchSharp.torch;
using static TorchSharp.torch.nn;

public sealed class TinyCnn : Module<Tensor, Tensor>
{
    private readonly Module<Tensor, Tensor> features;
    private readonly Module<Tensor, Tensor> pool;
    private readonly Module<Tensor, Tensor> classifier;

    public TinyCnn(long numClasses = 11) : base(nameof(TinyCnn))
    {
        features = Sequential(
            ("conv1", Conv2d(3, 16, kernel_size: 3, padding: 1, bias: false)),
            ("relu1", ReLU()),
            ("down1", MaxPool2d(kernel_size: 2)),
            ("conv2", Conv2d(16, 32, kernel_size: 3, padding: 1, bias: false)),
            ("relu2", ReLU()),
            ("down2", MaxPool2d(kernel_size: 2))
        );
        pool = AdaptiveAvgPool2d(new long[] { 1, 1 });
        classifier = Linear(32, numClasses);
        RegisterComponents();
    }

    public override Tensor forward(Tensor x)
    {
        using var scope = NewDisposeScope();
        x = features.forward(x);
        x = pool.forward(x);
        x = x.flatten(1);
        x = classifier.forward(x);
        return x.MoveToOuterDisposeScope();
    }
}

public static class Program
{
    public static void Main()
    {
        using var model = new TinyCnn(numClasses: 11);
        using var input = randn(new long[] { 4, 3, 64, 64 });
        using var logits = model.forward(input);

        Console.WriteLine($"[{string.Join(", ", logits.shape)}]");
        if (!logits.shape.SequenceEqual(new long[] { 4, 11 }))
            throw new InvalidOperationException("Unexpected logits shape");
    }
}
```

TorchSharp에서는 native tensor 메모리를 GC에만 맡기지 말고 `using`, `NewDisposeScope`, `MoveToOuterDisposeScope`를 사용해 수명을 명시적으로 관리해야 한다. 장시간 실행하는 ASP.NET 추론 서버에서 이를 놓치면 managed heap이 작아 보여도 native 메모리가 계속 증가할 수 있다.

## 14. 흔한 실패와 원인

### `expected input to have 3 channels, but got 64`

원인: HWC `(N,64,64,3)`를 NCHW로 오해했다.

해결:

```python
x = x.permute(0, 3, 1, 2).contiguous()
```

단, `permute`만 하면 RGB/BGR 색 순서는 바뀌지 않는다.

### `mat1 and mat2 shapes cannot be multiplied`

원인: Conv/Pool 뒤의 $H,W$를 잘못 계산해 `Linear(in_features, ...)`와 flatten 길이가 다르다.

해결 순서:

1. 각 층의 shape을 출력한다.
2. 출력 공식으로 예상값을 다시 계산한다.
3. 입력 해상도가 가변이면 `AdaptiveAvgPool2d((1,1))`를 classifier 앞에 둔다.
4. 빠른 실험에는 `LazyLinear`도 가능하지만, 첫 forward 전 파라미터 초기화·optimizer 생성·checkpoint 처리 차이를 이해한다.

### 학습은 되는데 검증 성능이 비정상적으로 높다

원인 후보: 같은 사람/연속 프레임/원본 이미지의 증강본이 train과 validation에 동시에 들어간 데이터 누수다.

해결: 파일 단위 무작위 분할보다 사람 ID, 촬영 세션, 영상 단위의 group split을 사용한다. 손동작 데이터에서는 같은 사용자의 인접 프레임이 매우 비슷할 수 있다.

### 결과가 실행마다 다르다

원인 후보: 초기 가중치, 데이터 순서, augmentation, DataLoader worker, CUDA convolution 알고리즘이 모두 난수나 비결정성을 포함한다.

해결: seed와 결정론 옵션을 제어하되, 성능 저하와 플랫폼 간 비동일성까지 실험 기록에 남긴다.

### 문제 4

## 16. 연습 문제 해설

### 해설 1

$$
H_{out}=\left\lfloor\frac{128+4-4-1}{2}+1\right\rfloor=64
$$

$$
W_{out}=\left\lfloor\frac{96+4-4-1}{2}+1\right\rfloor=48
$$

따라서 `(8, 32, 64, 48)`이다.

### 해설 2

파라미터:

$$
32(3\times5\times5+1)=2{,}432
$$

batch MACs:

$$
8\times64\times48\times32\times3\times5\times5
=58{,}982{,}400
$$

### 해설 3

- 시작: $J=1,R=1$
- 첫 Conv: $J=1,R=3$
- 둘째 Conv: $J=2,R=5$
- 셋째 Conv: $K_{eff}=2(3-1)+1=5$, 따라서 $J=2,R=5+(5-1)\times2=13$

최종 수용영역은 $13\times13$이다.

### 해설 4

```python
import cv2
import torch

bgr = cv2.imread("sample.jpg", cv2.IMREAD_COLOR)
if bgr is None:
    raise FileNotFoundError("sample.jpg")

rgb = cv2.cvtColor(bgr, cv2.COLOR_BGR2RGB)
x = torch.from_numpy(rgb).permute(2, 0, 1).contiguous()
x = x.to(dtype=torch.float32).div_(255.0).unsqueeze(0)
assert x.shape == (1, 3, 64, 64)
```

학습 때 mean/std 정규화를 사용했다면 추론에서도 같은 값과 순서로 추가 적용해야 한다.

## 17. 오늘의 핵심 요약

1. PyTorch `Conv2d`는 kernel을 뒤집지 않는 교차상관을 수행한다.
2. CNN은 지역 연결과 가중치 공유로 이미지의 공간 구조를 효율적으로 학습한다.
3. 출력 크기는 입력, padding, stride, dilation, kernel의 함수이며 classifier 설계 전에 반드시 계산해야 한다.
4. 파라미터 수가 작아도 큰 feature map에서는 MAC와 activation 메모리가 클 수 있다.
5. 수용영역은 깊이, stride, dilation에 따라 커지며 작은 물체 보존과 전체 문맥 확보 사이에 trade-off가 있다.
6. Python·C++·C#의 결과를 맞추려면 layout, RGB/BGR, dtype, 정규화, padding 규칙을 계약으로 고정해야 한다.
7. shape assertion과 수치 비교 테스트는 모델 구조 변경과 배포 포팅에서 가장 값싼 안전장치다.
